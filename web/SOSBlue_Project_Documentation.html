1 Executive Summary

SOSBlue is a native Android application enabling secure, offline, peer-to-peer communication using Bluetooth Low Energy (BLE) and Local Wi-Fi. It creates a decentralized mesh network allowing users to exchange text messages, share files, and communicate in groups — all without internet, cellular service, or SIM cards.

In disaster scenarios, network blackouts, remote areas, or large gatherings where traditional infrastructure fails, SOSBlue provides a reliable fallback. When connectivity is available, the app extends its reach through a Secure Cloud Relay.

Key Identity: Version 2.5.1  |  Package: com.antor.sosblue  |  Min Android 8.0 (API 26)  |  Target SDK: Android 15 (API 35)

3 Product Overview

3.1 Offline Layer (Primary)

Bluetooth Low Energy (BLE): Low-power communication with a device-to-device range of 15–100+ meters per hop. Messages automatically relay through intermediate devices (mesh network) to extend effective range far beyond a single hop.

Local Wi-Fi Network: High-bandwidth connection for file sharing and chat when devices are connected to the same Wi-Fi router or Hotspot.

3.2 Online Layer (Optional)

Secure Cloud Relay: When internet is available, messages are batched and relayed via secure cloud servers within ~30 seconds to reach distant users anywhere in the world.

Cloud Server: Handled by a robust backend architecture ensuring efficient message routing and delivery.

3.3 Hybrid Relay — No Message Lost

SOSBlue ensures zero message loss through its hybrid architecture. If a device is offline, encrypted messages are carried by nearby devices until they reach someone with internet access, who then relays them to the cloud. The message is queued and delivered as soon as the recipient comes online.

7 Security Architecture

Note: This section describes the security design philosophy without exposing implementation-level details.

7.1 Design Principles

#PrincipleImplementation
1Privacy by DesignNo personal info required. Users get random display IDs.
2Local-First StorageAll data stored exclusively on device. No central server.
3Encryption in TransitBLE/Wi-Fi messages encrypted to prevent interception.
4Encryption at RestLocal data secured via Android storage protection.
5No Cloud DependencyFully functional offline. Cloud is optional.

7.2 Security Features

FeatureDescription
Message EncryptionSHA-256 derived key encryption for all private and group chats — ensures confidentiality even over open BLE channels
Anonymous IdentityRandom 8-character display IDs, no personal data
User BlockingBlock specific users from messaging
Content FilteringConfigurable incoming message filters
No Backup by DesignPrevents unauthorized data extraction
ProGuard ObfuscationCode shrinking & obfuscation in release builds
Why is SOSBlue Encryption Powerful & Reliable? POWERFUL (SHA-256): SOSBlue uses SHA-256 to derive unique, high-entropy 256-bit keys from your specific Chat ID and Password. This cryptographic process ensures that even if a hacker intercepts the public BLE signals, the data appears as random noise. Without the exact credentials (which are never transmitted), the message content is mathematically infeasible to decrypt. RELIABLE (Offline): Security is 100% Offline & Decentralized. Keys are generated locally on your device and are never sent to a cloud server. This means there is no central "Key Server" to hack, block, or shut down. Your private communications remain secure and accessible even if the entire internet collapses.

8 Communication Protocols

8.1 BLE Communication Model

SOSBlue uses a connectionless BLE advertising/scanning model — no pairing, no connections, no handshakes. Messages are embedded directly in BLE advertisement service data packets and broadcast to all nearby devices.

8.2 Message Lifecycle

StepPhaseDescription
1CompositionUser composes message in chat interface
2CompressionPayload compressed (URL 5-bit/6-bit, Unicode optimization)
3EncryptionPrivate chats (Friend/Group) encrypted with shared key
4ChunkingData split into BLE-compatible chunks with binary headers
5AdvertisingEach chunk broadcast as BLE advertisement service data
6ScanningReceiver's Nordic Scanner detects and queues packets
7ReassemblyMessageProcessor reassembles chunks into complete message
8DecryptionPrivate messages decrypted with corresponding key
9DisplayUI updated via LiveData; message persisted in Room database
10Mesh RelayComplete messages re-broadcast for mesh range extension

8.3 Security Protocol

SOSBlue uses a custom binary packet protocol designed for efficiency and security. Packet structures are minimized to prevent pattern analysis, and payloads are encrypted using the methods described in the Security section.

Advanced Feature: JSON Power Links ---------------------------------- To share rich content, host a JSON file (e.g., "https://myjson.com/1.json") and send: "g//myjson.com/1.json". Format: { "message": "Long text message...", "images": [ {"thumb": "url...", "url": "url..."} ], "videos": [ {"length": "15:20", "url": "url..."} ] }

11.3 How SOSBlue Works — Network Scenarios

Core Promise: Even if the global Internet infrastructure collapses, SOSBlue allows you to stay connected and send messages to both known and unknown users nearby.

Scenario 1: Global Internet Shutdown (Fully Offline)

The entire Internet goes offline. With SOSBlue, you can still send messages to nearby users. Personal Chat and Group Chat are fully functional, with a device-to-device mesh range of 15–100+ meters, allowing messages to hop through multiple devices to extend reach.

Encrypted Relay: You can send encrypted messages to a specific person. Nearby users carry these messages automatically but cannot read them, ensuring privacy. Once the intended recipient's device comes within range of any carrier device, the message is automatically delivered. This feature has no range limitation — delivery depends on message propagation through nearby devices.

Scenario 2: You're Offline, Internet Available Globally

Even if your device has no Internet, SOSBlue uses nearby users' devices as relays. Encrypted messages are received by nearby devices but remain unreadable to them. If any relay device has Internet, your message is forwarded to the destination within ~30 seconds. If relay devices are also offline, the message is sent as soon as Internet becomes available. No message is lost.

Scenario 3: You're Online, Recipient is Offline

If your device is connected to the Internet but your friend's device isn't: your message is encrypted and queued on the server within ~30 seconds. Once the recipient comes online, the message is delivered immediately. This allows SOSBlue to function like a traditional messenger when Internet is available.

11.5 FAQ

QuestionAnswer
Why is there a ~30s delay for cloud relay?Intentional design to reduce server load. Messages are batched before sending. Real-time cloud messaging may be added in future updates.
Can relay carriers read my messages?No. Messages are encrypted before transmission. Carrier devices forward the data without being able to decrypt it.
Does SOSBlue replace emergency services?No. SOSBlue is a communication tool, not a replacement for official emergency services. Always contact authorities first.
Important: Never assume the Internet will always be available. Keep SOSBlue installed — it could be essential in emergencies. Report bugs via sosblueapp@gmail.com.
